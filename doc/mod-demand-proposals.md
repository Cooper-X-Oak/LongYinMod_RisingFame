# 社区需求与功能提案

更新时间：`2026-03-25`

## 01. 目的

这份文档不是为了罗列“什么都能做”，而是要先回答两个更现实的问题：

1. 玩家现在最强烈、最稳定的需求是什么
2. 在《龙胤立志传》当前环境下，什么方向最适合先做成稳定版

当前项目的前提仍然不变：

- 稳定优先
- 不额外增加明显计算量
- 尽量避免复杂 UI 注入与高风险 Hook
- 优先做“安装后就能感知到收益”的功能

<a id="demand-categories"></a>
## 03. 需求归类

### 一、减肝 / 效率提升是最稳定的高频需求

这是当前最明确的一类需求，而且不是“少数硬核玩家”的偏好，而是被多个方向共同验证出来的：

- Steam 社区里，玩家频繁讨论成长慢、练功节奏、门派经营链路过长、部分系统耗时偏重
- Steam 社区主页上已经出现“护肝体验”“自动买卖”“战斗中修炼获取经验”等大型体验修改指南
- Bilibili 上，《龙胤立志传》Demo 阶段就出现了通过修改月份限制来对冲试玩时长压力的内容
- Bilibili 的“省时 MOD / 护肝功能 / 修改器”类视频里，关键词高度集中在：
  - 自动鉴宝 / 自动扫货 / 自动买卖
  - 刷新拍卖 / 刷新突破词条 / 黑市与锻造刷新
  - 读书练功重复时间与效率问题
  - 经验倍率 / 护肝功能 / 菜单修改器
- 官方自 `2026-03-19` 起连续发布更新日志，并在 `高频问题和Bug答疑。后续优化计划` 讨论串里持续回应体验问题，方向集中在“快时间流逝”“加快小游戏节奏”“补提示”“增加直接交谈”等改动上，说明开发组自己也在处理节奏与操作负担问题

结论：

- “减肝”不是外挂式需求，而是和官方优化方向一致的真实痛点
- 这类需求最适合先做，因为玩家收益直观，且你现有 mod 的方向已经和它高度重合
- 但 B 站二次样本也说明，`P0` 不能只理解成“经验倍率”，还应包括刷新类、自动购买类、读书练功减重复、快捷键体验与安装稳定性

### 二、交互提效和反馈清晰度也非常高频

除了数值本身，玩家另一个痛点是：

- 流程长
- 结算反馈弱
- 某些系统看起来“像生效了”，但玩家感知不到
- 一些门派 / 官府 / 学习链路中，提示和跳转不够直观

官方更新里已经补了：

- 地图按钮提示
- 门派详情显示
- 结局提示
- 直接交谈入口
- 商店折扣数值

这些都说明一个事实：

- 很多问题不是“内容不够”，而是“信息反馈不够明确”

这对 mod 来说是好消息，因为：

- 原生提示增强往往比大功能重做更稳
- 也更符合“不增加明显计算量”的要求

### 三、安装兼容 / 框架稳定性是实际门槛，不是附属问题

社区里已经有两个很直接的信号：

- 有玩家在 Steam 留言提到黑屏、进不去、云存档相关问题
- 英文机翻指南明确写了“BepInEx 似乎不可用，因此改用 MelonLoader + XUnity.AutoTranslator”
- Bilibili 评论区也出现了大量“安装失败 / 依赖下载失败 / GitHub 需要代理 / cpp2il 报错 / MelonLoader Il2CppAssemblies 目录问题”等反馈

这不一定说明 BepInEx 一定不行，但至少说明：

- 不同机器环境下，mod 框架兼容性并不稳定
- “固定版本 + 固定架构 + 傻瓜安装方式 + 支持边界”本身就是高价值交付

对当前项目的意义是：

- 后续如果公开分发，安装体验必须被当成正式功能的一部分
- 不能只看代码是否能跑，也要看用户能否无痛装上

### 四、外观 / 立绘 / 角色美化在同类作品里非常高频

Bilibili 上《英雄立志传：三国志》的 mod 生态非常清楚：

- 立绘替换
- 肖像扩充
- 外观修正
- 懒人包合集

这说明：

- 武侠 / 养成 / 角色关系类游戏，外观资源一定会形成高频 mod 需求
- 但这类内容偏资源生产，不是当前仓库最合适的第一优先级

当前更适合把它放在后续规划里，而不是压到第一波稳定版里。

### 五、剧情 / 内容拓展也有需求，但不适合做首发方向

《英雄立志传：三国志》和《大侠立志传》的 B 站 mod 内容里，剧情扩展非常活跃：

- 新事件
- 新人物线
- 内容拓展
- 大型剧情章节

这说明：

- 如果一个立志传类游戏建立了足够大的玩家基数，内容 mod 会自然增长
- 但内容扩展开发成本、测试成本、兼容成本都更高

对当前项目来说：

- 这类方向更适合放在 P2/P3 之后
- 不适合作为你当前“稳定、轻量、护肝”的主线仓库目标

<a id="demand-priority"></a>
## 04. 当前最值得做的提案优先级

### P0：低风险减肝 / 交互提效包

这是最适合当前仓库继续深挖的方向。

核心定义：

- 用最少 Hook 数量
- 做最直接可感知的收益
- 不引入额外高频逻辑
- 优先沿用游戏原生流程和原生提示

建议包含的内容边界：

- 经验 / 功绩 / 好感 / 抄书等低风险倍率与效率增强
- 读书、修炼、技艺、门派 / 官府相关流程中的轻量提示增强
- 复用原生控制器里的 `Auto...`、`Sure...`、`Finish...` 能力做轻量交互提效
- 不做自绘大 UI，不做组件注入，不做复杂后台逻辑

从 B 站评论信号来看，P0 里最具共识的“省时类诉求”大致包括：

- 自动鉴宝、刷新拍卖、刷新突破词条
- 黑市 / 锻造 / 藏经阁等高成本刷新类操作
- 自动扫货 / 自动购买（材料、珍宝、酒馆加属性）
- 快捷键自定义（ALT 键冲突与不顺手）

并且还要强调一件事：

- 便利功能最好和强倍率拆层
- 允许玩家只开 QoL，不开明显破坏平衡的倍率项

为什么它是 P0：

- 与你当前项目基因完全一致
- 技术风险最低
- 用户收益最明确
- 与官方近期优化方向一致
- 也最容易写清楚 README 与支持边界

### P1：原生消息增强 / 可视化反馈包

这是 P0 的自然延伸，但可以晚一步做。

目标：

- 让玩家更明确地知道“什么时候加成生效了”“加了多少”“属于哪一类收益”
- 复用原生消息、飘字、武学经验展示，而不是自己造一整套 UI

适合内容：

- 经验结算提示
- 功绩 / 好感变化提示
- 学习与抄书结果提示
- 轻量世界消息或个人消息

之所以放 P1 而不是单独压成首发主线，是因为：

- 它很有价值
- 但如果做得太早，容易分散掉对“真实收益稳定性”的验证精力

### P2：安装兼容包 / 固定运行环境

这是分发层面的高优先级，不是功能层面的高优先级。

建议方向：

- 固定 BepInEx 版本
- 固定架构与目录结构
- 固定推荐安装步骤
- 写清楚不支持的边界
- 增加 QA 检查表

为什么重要：

- 这类项目最大的口碑风险，很多时候不是功能不好，而是“装不上”“开不了”“报错后不知道怎么排查”

### P3：外观 / 立绘 / 角色表现层扩展

适合在基础版本稳定后，再看是否拆成独立资源仓库或额外分支。

### P4：剧情 / 门派 / 自建势力等内容型扩展

这类方向最吸引人，但技术和维护成本也最高。

建议：

- 不放进当前轻量核心版主仓库
- 如果未来真要做，最好单开支线或单独项目

<a id="demand-p0-scope"></a>
## 05. P0 细化定义

为了避免提案名字太虚，这里把 P0 再收窄成一句话：

> 以“少 Hook、低开销、强感知”为原则，做一个围绕成长效率与原生反馈增强的轻量体验包。

P0 推荐包含的三条子线：

1. 真实入账层的低风险倍率增强
2. 玩家学习 / 读书 / 门派功绩流程的轻量提效
3. 原生通知与结算反馈增强

P0 明确不做：

- 运行时大范围对象扫描
- 注入 Unity 组件
- 高频日志
- 自定义大型 UI 框架
- 大型剧情扩写
- 复杂 AI 或全局行为重写

<a id="demand-roadmap"></a>
## 06. 与当前仓库最匹配的落地顺序

建议顺序：

1. 先守住已有稳定核心 Hook
2. 再补 P0 的原生提示与流程提效研究
3. 最后再考虑是否拆出“玩家专用分支”或“安装兼容分支”

原因很简单：

- 当前最有价值的不是把功能做多，而是把“这东西装上就有效，而且不容易崩”做稳

## 07. 结论

如果只选一个方向继续往下挖，当前最值得作为 `P0` 的，不是外观，不是剧情，也不是大而全整合包，而是：

**低风险减肝 / 交互提效包**。

它兼顾了：

- 社区真实痛点
- 现有代码积累
- 你的稳定性要求
- 用户装上后的即时收益
- 后续扩展到原生提示与玩家流程分支的可能性
